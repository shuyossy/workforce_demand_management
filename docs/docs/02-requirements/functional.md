# 機能要件

## 題材

社内業務を模した **休暇申請ワークフロー**。3 状態（PENDING / APPROVED / REJECTED）の最小ワークフロー。社員番号・所属・職層に基づく承認権限ロジックを通じて、社内ライブラリ準拠 IF（`UserDto` / `MSTBusinessException` / `@InjectLogger`）が自然に活きる構成。

## ユースケース一覧

| ID                  | 概要                                                           | 主アクター |
| ------------------- | -------------------------------------------------------------- | ---------- |
| ApplyLeaveUseCase   | 休暇種別 / 期間 / 理由を入力して申請                           | 申請者     |
| ApproveLeaveUseCase | PENDING の他人の申請に対し承認                                 | 承認権者   |
| RejectLeaveUseCase  | PENDING の他人の申請に対し却下（コメント必須）                 | 承認権者   |
| ListLeavesUseCase   | 自分の申請（MINE）と承認候補（APPROVABLE）一覧をタブで切替表示 | 全社員     |
| FindLeaveUseCase    | 任意の申請詳細を取得（他人の申請も参照可）                     | 全社員     |

## ユーザストーリー

| ID    | ストーリー                                                                                                                         |
| ----- | ---------------------------------------------------------------------------------------------------------------------------------- |
| US-01 | 一般従業員として、休暇種別・期間・理由を入力して休暇を申請したい。そうすれば事前に休みを取得する旨を組織に伝えられる。             |
| US-02 | 管理職として、自組織のメンバーから上がってきた休暇申請を承認したい。そうすれば部下の休暇取得を許可できる。                         |
| US-03 | 管理職として、自組織のメンバーから上がってきた休暇申請を理由付きで却下したい。そうすれば不適切な申請を理由とともに差し戻せる。     |
| US-04 | 一般従業員として、自分の申請状況および自分が判断すべき申請を一覧で確認したい。そうすれば日々の業務の中で漏れなく状況を把握できる。 |
| US-05 | 全社員として、特定の申請の詳細を確認したい。そうすれば判断や事実確認に必要な背景情報を取得できる。                                 |

## 受け入れ基準

ユースケースごとに Given-When-Then 形式で記述する。文中の業務エラー文言は §業務エラーシナリオ で参照されているメッセージと一致させる。

### ApplyLeaveUseCase

- **AC-APPLY-01**: Given ログイン済みかつ休暇種別・期間・申請理由のすべてが妥当 / When 申請を確定する / Then 申請が「申請中」状態で受け付けられ、受付完了が提示され、申請一覧に当該申請が表示される。
- **AC-APPLY-02**: Given 開始日が終了日より後 / When 申請を確定する / Then 入力エラー「開始日は終了日より前である必要があります」が提示され、申請は受け付けられない。
- **AC-APPLY-03**: Given 申請理由が空または 500 文字を超える / When 申請を確定する / Then 該当項目の入力エラーが提示され、申請は受け付けられない。
- **AC-APPLY-04**: Given 休暇種別・開始日・終了日のいずれかが未入力 / When 申請を確定する / Then 必須項目エラーが提示され、申請は受け付けられない。

### ApproveLeaveUseCase

- **AC-APPROVE-01**: Given 自分以外の同一組織メンバーが申請した「申請中」の申請があり、自分が管理職階層に属する / When 承認を確定する / Then 申請の状態が「承認済」に遷移し、承認権者の社員番号・氏名・判断日時・任意のコメントが保全される。
- **AC-APPROVE-02**: Given 対象申請が既に「承認済」または「却下済」 / When 承認を確定する / Then 業務エラー「申請中の休暇申請のみ承認/却下できます」が提示され、申請の状態は変更されない。
- **AC-APPROVE-03**: Given 対象申請が自分自身の申請 / When 承認を確定する / Then 業務エラー「自分の申請は承認/却下できません」が提示され、申請の状態は変更されない。
- **AC-APPROVE-04**: Given 自分の所属組織が申請者と異なる、または自分が管理職階層に属さない / When 承認を確定する / Then 業務エラー「この休暇申請を承認/却下する権限がありません」が提示され、申請の状態は変更されない。
- **AC-APPROVE-05**: Given 指定 ID の申請が DB に存在しない / When 承認を確定する / Then 業務エラー「指定された休暇申請が存在しません（id=...）」が提示され、申請の状態は変更されない。

### RejectLeaveUseCase

- **AC-REJECT-01**: Given 自分以外の同一組織メンバーが申請した「申請中」の申請があり、自分が管理職階層に属し、却下コメントが 1〜500 文字 / When 却下を確定する / Then 申請の状態が「却下済」に遷移し、判断者情報と却下コメントが保全される。
- **AC-REJECT-02**: Given 却下コメントが空または 500 文字を超える / When 却下を確定する / Then 入力エラーが提示され、申請の状態は変更されない。
- **AC-REJECT-03 〜 05**: ApproveLeaveUseCase の AC-APPROVE-02〜05 と同等の挙動（状態違反 / 自己却下 / 権限なし / 不在）。

### ListLeavesUseCase

- **AC-LIST-01**: Given ログイン済み、スコープ「自分の申請」 / When 一覧を取得する / Then 自分が申請したすべての申請（状態を問わず）が申請日時降順で提示される。0 件の場合は空一覧として提示される。
- **AC-LIST-02**: Given ログイン済み、スコープ「承認候補」 / When 一覧を取得する / Then 自分と同一組織で「申請中」かつ申請者が自分以外の申請のみが申請日時昇順で提示される。0 件の場合は空一覧として提示される。
- **AC-LIST-03**: Given 「承認候補」一覧に表示されている申請 / When 詳細から承認を試みる / Then 承認可否は ApprovalPolicy で改めて判定され、権限がなければ業務エラーが提示される（一覧表示自体は承認権限を保証しない）。

### FindLeaveUseCase

- **AC-FIND-01**: Given 任意の申請の ID（自分の申請でも他人の申請でも可） / When 詳細を取得する / Then 申請者・休暇種別・期間・申請理由・状態・判断情報・利用者の承認可否フラグが提示される。
- **AC-FIND-02**: Given 指定 ID の申請が DB に存在しない / When 詳細を取得する / Then 業務エラー「指定された休暇申請が存在しません（id=...）」が提示される。
- **AC-FIND-03**: Given 利用者が承認権者かつ申請が「申請中」 / When 詳細を表示する / Then 承認 / 却下ボタンが操作可能な状態で提示される。それ以外の場合は操作ボタンは非表示。

## 業務エラーシナリオ

業務エラーは `MSTBusinessException` として送出され、JSF の `FacesMessage` で画面に提示される。`messages.properties` の `RCB00101-W` を除き、本サンプルではメッセージ文字列を例外コンストラクタに直接指定する方式を採る（社内ライブラリ置換時は ErrorCode 化される想定）。

| シナリオ ID | シナリオ                  | 発生条件                                                             | システム挙動 (画面提示メッセージ)                                | 関連 BR                |
| ----------- | ------------------------- | -------------------------------------------------------------------- | ---------------------------------------------------------------- | ---------------------- |
| ES-01       | 対象申請が存在しない      | 承認・却下・詳細表示時に指定 ID の申請が DB に無い                   | 「指定された休暇申請が存在しません（id=...）」                   | BR-PRE-005             |
| ES-02       | 状態違反                  | 承認・却下時に対象申請が既に APPROVED または REJECTED                | 「申請中の休暇申請のみ承認/却下できます（現在の状態=...）」      | BR-PRE-002, BR-INV-002 |
| ES-03       | 自己承認 / 自己却下       | 承認・却下時に自分自身が申請者                                       | 「自分の申請は承認/却下できません」                              | BR-PRE-001             |
| ES-04       | 権限なし                  | 承認・却下時に所属組織不一致または管理職階層に属さない               | 「この休暇申請を承認/却下する権限がありません」                  | BR-POL-001             |
| ES-05       | 期間順序違反（入力時）    | 申請時に開始日が終了日より後                                         | 「開始日は終了日より前である必要があります」 (`RCB00101-W` 由来) | BR-INV-001             |
| ES-06       | 必須項目欠落 / 文字数超過 | 申請時に必須項目欠落・申請理由が範囲外、却下時に却下コメントが範囲外 | Bean Validation 既定メッセージ（項目ごと）                       | BR-PRE-003, BR-PRE-004 |
| ES-07       | 認証セッション切れ        | 任意の操作時にセッションが失効                                       | `/login.xhtml` へリダイレクト                                    | -                      |

## 画面構成と遷移

| 画面     | パス                       | 役割                                     |
| -------- | -------------------------- | ---------------------------------------- |
| ログイン | `/login.xhtml`             | dev ダミーユーザを選択してセッション確立 |
| 申請一覧 | `/leaves/list.xhtml`       | MINE / APPROVABLE タブ                   |
| 申請新規 | `/leaves/new.xhtml`        | 休暇種別 / 期間 / 理由 を入力            |
| 申請詳細 | `/leaves/detail.xhtml?id=` | 承認/却下ボタン（rendered で制御）       |
| エラー   | `/error.xhtml`             | `<error-page>` の着地                    |

遷移：ログイン → 一覧 → 新規 or 詳細 → 一覧へ戻る。

## 認証要件

- `AuthenticationFilter` がセッション内 `UserInfoContext.getUser()` の非 null を確認
- 未認証は `/login.xhtml` へリダイレクト（skipPaths: ログイン画面 / `jakarta.faces.resource` / `resources` / エラーページ）
- 開発用：`DevLoginAuthenticationAdapter`（ダミーユーザリストから UserDto を組み立て `UserInfoContext` に格納）
- 将来：`HeaderAuthenticationAdapter`（リクエストヘッダ `X-User-Id` から組み立て）
- `APP_AUTH_MODE` で切り替え可能（差し替え可能設計）

## スコープ外

本サンプルアプリは Jakarta EE ボイラープレートの最小ユースケース実証を目的とするため、以下は意図的にスコープ外とする。実プロジェクト適用時には必要に応じて追加すること。

- 休暇残日数の管理（種別ごとの付与・消化・繰越）
- 半休 / 時間休
- 申請取消（撤回）・再申請ワークフロー
- 複数階層の承認フロー（係長 → 課長 → 部長 等のチェーン）
- メール / チャット通知
- カレンダー連携（出退勤・在席情報）
- 監査ログの長期保全と検索 UI
- 多言語対応 / アクセシビリティ要件
