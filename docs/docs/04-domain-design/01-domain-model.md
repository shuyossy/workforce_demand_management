# ドメインモデル

本章はドメイン層のモデル（エンティティ・値オブジェクト・集約・ドメインサービス）を 1 件ずつ構造化して記述する。新規プロジェクトに本ボイラープレートを適用した際は、本章のテンプレ（§記載方法の雛形）をそのまま流用し、案件のドメインを記述する。サンプル例（タスク管理）は本章末尾に記載してあるため、不要であれば §サンプル例 を削除する（→ [サンプル削除ガイド](../getting-started/strip-sample)）。

## 記載方法の雛形

ボイラープレート適用先のチームは、新しいドメインモデルを追加するたびに以下のスケルトンをコピーして埋める。

### 共通ルール

- **識別子は実装クラス名と完全一致させる**（英語、PascalCase）。ドメイン設計とコードの 1:1 対応をトレーサビリティの起点とするため。
- **記述は業務目線で書く**。フレームワークやインフラ詳細（`@ApplicationScoped`、JPA、PrimeFaces、HTTP、SQL 等）はドメイン設計には書かない（必要なら技術設計書側に置く）。
- **不変条件は『常に守られるべき状態ルール』を業務目線で列挙**する。技術的バリデーション（NotNull アノテーション等）の話ではなく、業務上常に真であるべき条件を書く。
- **振る舞いは業務的に何が起きるかを書く**。実装シグネチャの再掲ではなく、業務操作の意図と前提・効果を述べる。
- ルール ID（`BR-INV-***` / `BR-POL-***` 等）と相互参照する場合は `(BR-XXX-NNN)` で括弧書き付記する。

### エンティティ / 値オブジェクト / 集約ルート 用スケルトン

```
### <識別子>

- 識別子: <実装クラス名 / 英語 PascalCase>
- 種類: <エンティティ / 値オブジェクト / 集約ルート のいずれか>
- 不変条件
  - <常に守られるべき状態ルールを業務目線で列挙>
- 属性
  - <属性名>: <業務的な意味（型ではなく役割を書く）>
- 振る舞い
  - <操作名>: <業務的に何が起きるか。前提と効果を簡潔に>
- その他
  - <ライフサイクル / 同一性の判定基準 / 関連エンティティとの関係など必要に応じて>
```

### ドメインサービス 用スケルトン

```
### <識別子>

- 識別子: <実装クラス名 / 英語 PascalCase>
- 前提条件
  - <この判定/操作を呼び出す前に成立していなければならない条件>
- 入力: <パラメータの業務的内容を簡潔に列挙>
- 出力: <返却内容を簡潔に>
- メインフロー
  1. <番号付き箇条書き、業務目線>
- 代替フロー
  a. <代替条件>
     1. <番号付き箇条書き>
- 例外
  a. <例外条件>
     - <システム挙動を箇条書き>
- 事後処理
  - <呼び出し後に副作用として実施する業務処理。なければ「なし」>
```

## サンプル例

:::note サンプル例
本セクションはタスク管理サンプルの記載例。適用先プロジェクトで利用しない場合は本セクションごと削除する（→ [サンプル削除ガイド](../getting-started/strip-sample)）。
:::

### Task

- 識別子: Task
- 種類: 集約ルート
- 不変条件
  - タイトルは非空白である。
  - 完了操作は未完了（TODO）状態からのみ実行できる。既に完了（DONE）の Task を再度完了させることはできない。
  - 完了日時は完了状態のときのみ設定される（未完了時は null）。
- 属性
  - id: 永続化識別子（新規作成時は未採番）。
  - title: タスクの内容を表す表題。
  - status: 現在の状態（未完了 / 完了）。
  - createdAt: 作成日時。
  - completedAt: 完了日時（未完了時は無し）。
- 振る舞い
  - create: タイトルと作成日時を受け取り、未完了状態の新規 Task を生成する。
  - complete: 未完了状態の Task を完了状態へ遷移させ、完了日時を記録する。既に完了状態の場合は実行できない（設計上到達しない防御であり、ユースケース側で事前に状態を再評価し業務エラーへ翻訳する）。
- その他
  - 同一性は識別子（id）で判定する。永続化前（新規作成直後）は id を持たない。
