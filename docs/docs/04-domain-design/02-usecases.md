# ユースケース

本章は業務のユースケースを 1 件ずつ構造化して記述する。視点はあくまで業務（アクターとシステムの振る舞い）であり、フレームワーク・実装クラス名・例外コード・データベース列名等は本章には書かない。新規プロジェクトに本ボイラープレートを適用した際は、本章のテンプレ（§記載方法の雛形）をそのまま流用し、案件のユースケースを記述する。サンプル例（タスク管理）は本章末尾に記載してあるため、不要であれば §サンプル例 を削除する（→ [サンプル削除ガイド](../getting-started/strip-sample)）。

## 記載方法の雛形

ボイラープレート適用先のチームは、新しいユースケースを追加するたびに以下のスケルトンをコピーして埋める。

### 共通ルール

- **識別子は実装クラス名と完全一致させる**（英語、PascalCase、語尾は `UseCase`）。ユースケース設計と入力 port（`application.port.in`）の 1:1 対応をトレーサビリティの起点とするため。
- **メインフロー / 代替フロー / 例外はアクター視点の番号付き箇条書きで書く**。シーケンス図、コンポーネント名（Bean / Service / Repository / Port 等）、フレームワーク用語は本章には書かない。
- **事後処理は『業務アウトカム（操作の結果として業務上何が成立するか）』を書く**。データベースカラムへの値保存・ログ出力・トランザクション境界等の技術スナップショット表現は禁止。事後条件（postcondition）は本ボイラープレートでは管理しない方針（§事後条件についての方針）。
- 業務ルール（BR-INV-/BR-POL-/BR-PRE-）を引用する場合は `(BR-XXX-NNN)` で括弧書き付記する。
- 入力 / 出力は **業務的に何を受け取り何を返すか** を簡潔に列挙する。コマンドクラス名や DTO 型名は書かない。

### 事後条件についての方針

ユースケース設計時点では「事後条件（postcondition、操作後に必ず成立しているべき状態）」は明示的に管理しない。理由は以下のとおり:

- 状態の整合性は **不変条件（ドメインモデル §エンティティ / 値オブジェクトの不変条件 参照）** と **前提条件（BR-PRE-\*）** で十分担保できる。
- 「列に何を保存する」レベルの事後条件は実装詳細にすぎず、業務目線のドキュメントに紛れ込ませると認知負荷を上げる。
- 業務上有意な「結果として何が起きるか」は本テンプレートの **事後処理（業務アウトカム）** で表現する（例: 「業務オブジェクトが受付状態になり一覧に表示される」「操作時点のアクター情報が監査用に保全される」）。

### ユースケース 用スケルトン

```
### <識別子>

- 識別子: <実装クラス名 / 英語 PascalCase / 語尾 UseCase>
- アクター: <主アクター（業務上の役割名）>
- 前提条件
  - <この操作を開始できる業務上の条件>
- 入力: <受け取る業務情報を簡潔に列挙>
- 出力: <返却する業務情報を簡潔に>
- メインフロー
  1. <アクター・システムの振る舞いを業務目線で番号付き箇条書き>
- 代替フロー
  a. <代替条件>
     1. <番号付き箇条書き>
- 例外
  a. <例外条件>
     - <システム挙動を箇条書き（業務エラーの提示、操作の中断 等）>
- 事後処理
  - <業務アウトカム（操作の結果として業務上何が成立するか）。なければ「なし」>
```

## サンプル例

:::note サンプル例
本セクションはタスク管理サンプルの記載例。適用先プロジェクトで利用しない場合は本セクションごと削除する（→ [サンプル削除ガイド](../getting-started/strip-sample)）。
:::

### ListTasksUseCase

- 識別子: ListTasksUseCase
- アクター: 利用者
- 前提条件
  - 特になし（本サンプルは認証を持たず、全利用者が同一のタスク一覧を参照する）。
- 入力: なし
- 出力: タスクの一覧（識別子・タイトル・状態・作成日時・完了日時）
- メインフロー
  1. 利用者が一覧画面を開く。
  2. システムが全タスクを作成日時の降順で取得する。
  3. システムが一覧を提示する。
- 代替フロー
  a. タスクが 1 件も存在しない場合
  1.  システムが空の一覧を提示する。
- 例外
  - なし
- 事後処理
  - なし（読み取り専用）

### CreateTaskUseCase

- 識別子: CreateTaskUseCase
- アクター: 利用者
- 前提条件
  - 特になし。
- 入力: タイトル
- 出力: なし（作成後は一覧画面へ遷移）
- メインフロー
  1. 利用者がタイトルを入力し作成を指示する。
  2. システムが未完了（TODO）状態の新規タスクを作成日時とともに登録する。
  3. システムが作成完了を提示し、一覧画面へ遷移する。
- 代替フロー
  - なし
- 例外
  a. タイトルが未入力または上限文字数を超える場合
  - 入力エラーを提示し、タスクは作成されない。
- 事後処理
  - 作成されたタスクが一覧の先頭（作成日時降順）に表示されるようになる。

### CompleteTaskUseCase

- 識別子: CompleteTaskUseCase
- アクター: 利用者
- 前提条件
  - 完了対象のタスクが存在する。
- 入力: 対象タスクの識別子
- 出力: なし（完了後は一覧画面へ遷移）
- メインフロー
  1. 利用者が一覧から対象タスクの完了を指示する。
  2. システムが対象タスクが未完了（TODO）であることを確認する（BR-PRE-001）。
  3. システムがタスクを完了（DONE）状態に遷移させ、完了日時を記録する。
  4. システムが一覧画面へ遷移し、完了後の状態を提示する。
- 代替フロー
  - なし
- 例外
  a. 対象タスクが既に完了（DONE）状態の場合
  - 業務エラー「既に完了したタスクです」を提示し、現在の一覧画面に留置する（回復可）。ユーザ操作（同一タスクの二重完了）で発生し得るため回復可能な業務エラーとして扱う。
    b. 対象タスクの識別子が存在しない場合
  - 削除機能を持たない本サンプルでは通常の操作では発生し得ない想定外状態（POST 偽装・データ不整合）のため、回復不可の業務エラーとしてエラー画面へ遷移する（BR-PRE-002）。
- 事後処理
  - 対象タスクが一覧上で完了状態として表示され、完了ボタンが表示されなくなる。
