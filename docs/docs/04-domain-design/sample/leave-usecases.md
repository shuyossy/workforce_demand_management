## ApplyLeaveUseCase

- 識別子: ApplyLeaveUseCase
- アクター: 申請者（一般従業員）
- 前提条件
  - 申請者がログイン済みで、社員番号 / 氏名 / 主務所属が確定している。
- 入力: 休暇種別, 開始日, 終了日, 申請理由
- 出力: 受け付けた申請の識別子
- メインフロー
  1. 申請者が休暇種別・期間・申請理由を入力する。
  2. システムが入力値の妥当性を確認する（必須項目の欠落がないこと、開始日が終了日以前であること（BR-INV-001）、申請理由が 1〜500 文字に収まること（BR-PRE-003））。
  3. システムが申請を「申請中」状態として受け付ける。
  4. システムが申請者に受付完了を提示し、申請一覧へ誘導する。
- 代替フロー
  - なし
- 例外
  a. 入力値が妥当でない場合
  - 該当項目の入力エラーを申請者に提示し、申請は受け付けない。
- 事後処理
  - 受け付けた申請が「申請中」状態として申請者本人の一覧および同一組織内の承認権者の承認待ち一覧から参照可能になる。
  - 申請時点の申請者所属情報が監査用に保全される（将来の組織異動後も申請当時の所属を辿れる）。

## ApproveLeaveUseCase

- 識別子: ApproveLeaveUseCase
- アクター: 承認権者（同一組織の管理職）
- 前提条件
  - 承認権者がログイン済みで、社員番号 / 主務所属 / 階層が確定している。
  - 対象申請が事前に特定されている（申請一覧または詳細画面からの導線）。
- 入力: 申請の識別子, 承認コメント（任意）
- 出力: なし（処理結果は申請の状態として表現される）
- メインフロー
  1. 承認権者が対象申請を選択し承認を指示する。
  2. システムが対象申請の存在を確認する（BR-PRE-005）。
  3. システムが対象申請の現在状態が「申請中」であることを確認する（BR-PRE-002）。
  4. システムが承認権者が申請者本人でないことを確認する（BR-PRE-001）。
  5. システムが承認権者に承認権限があることを確認する（同一組織かつ管理職階層に属する。BR-POL-001）。
  6. システムが対象申請を「承認済」へ遷移させ、判断者情報を確定する。
  7. システムが承認権者に処理完了を提示し、一覧へ戻る。
- 代替フロー
  - なし
- 例外
  a. 対象申請が存在しない場合
  - 業務エラー「指定された休暇申請が存在しません」を提示し、申請の状態は変更しない。
    b. 対象申請が「申請中」でない場合
  - 業務エラー「申請中の休暇申請のみ承認/却下できます」を提示し、申請の状態は変更しない。
    c. 承認権者が申請者本人の場合
  - 業務エラー「自分の申請は承認/却下できません」を提示し、申請の状態は変更しない。
    d. 承認権者に承認権限がない場合
  - 業務エラー「この休暇申請を承認/却下する権限がありません」を提示し、申請の状態は変更しない。
- 事後処理
  - 対象申請が「承認済」状態になり、申請者本人および承認権者の双方の一覧で確定状態として参照可能になる。
  - 判断時点の判断者情報（社員番号・氏名・判断日時・コメント）が監査用に保全される。

## RejectLeaveUseCase

- 識別子: RejectLeaveUseCase
- アクター: 承認権者（同一組織の管理職）
- 前提条件
  - 承認権者がログイン済みで、社員番号 / 主務所属 / 階層が確定している。
  - 対象申請が事前に特定されている。
- 入力: 申請の識別子, 却下コメント（必須・1〜500 文字。BR-PRE-004）
- 出力: なし（処理結果は申請の状態として表現される）
- メインフロー
  1. 承認権者が対象申請を選択し却下コメントを入力する。
  2. システムが対象申請の存在を確認する（BR-PRE-005）。
  3. システムが対象申請の現在状態が「申請中」であることを確認する（BR-PRE-002）。
  4. システムが承認権者が申請者本人でないことを確認する（BR-PRE-001）。
  5. システムが承認権者に承認権限があることを確認する（BR-POL-001）。
  6. システムが対象申請を「却下済」へ遷移させ、判断者情報と却下コメントを確定する。
  7. システムが承認権者に処理完了を提示し、一覧へ戻る。
- 代替フロー
  - なし
- 例外
  a. 却下コメントが空または 500 文字を超える場合
  - 入力エラーを提示し、申請の状態は変更しない。
    b. 対象申請が存在しない場合
  - 業務エラー「指定された休暇申請が存在しません」を提示し、申請の状態は変更しない。
    c. 対象申請が「申請中」でない場合
  - 業務エラー「申請中の休暇申請のみ承認/却下できます」を提示し、申請の状態は変更しない。
    d. 承認権者が申請者本人の場合
  - 業務エラー「自分の申請は承認/却下できません」を提示し、申請の状態は変更しない。
    e. 承認権者に承認権限がない場合
  - 業務エラー「この休暇申請を承認/却下する権限がありません」を提示し、申請の状態は変更しない。
- 事後処理
  - 対象申請が「却下済」状態になり、申請者本人および承認権者の双方の一覧で確定状態として参照可能になる。
  - 判断時点の判断者情報（社員番号・氏名・判断日時・却下コメント）が監査用に保全される。

## ListLeavesUseCase

- 識別子: ListLeavesUseCase
- アクター: 全社員（申請者・承認権者を区別しない）
- 前提条件
  - 利用者がログイン済みで、社員番号 / 主務所属が確定している。
- 入力: 一覧スコープ（自分の申請 / 承認候補のいずれか）
- 出力: 申請の一覧（識別子・申請者・休暇種別・期間・状態・申請日時）
- メインフロー
  1. 利用者が「自分の申請」または「承認候補」のスコープを選択する。
  2. システムが選択スコープに応じた申請一覧を取得する。
     - 「自分の申請」スコープ: 利用者自身が申請したものすべて（状態を問わず）を申請日時降順で返す。
     - 「承認候補」スコープ: 利用者と同一組織で「申請中」かつ申請者が自分以外のものを申請日時昇順で返す。
  3. システムが一覧を利用者に提示する。
- 代替フロー
  - なし
- 例外
  - なし（一覧が 0 件の場合は空一覧として提示する）
- 事後処理
  - 一覧の取得は参照系であり申請の状態は変更されない。
  - 注: 「承認候補」一覧に含まれることは承認権限を最終的に保証しない。最終的な承認可否は ApproveLeaveUseCase / RejectLeaveUseCase 実行時に ApprovalPolicy で改めて判定される（BR-POL-001）。

## FindLeaveUseCase

- 識別子: FindLeaveUseCase
- アクター: 全社員
- 前提条件
  - 利用者がログイン済みで、社員番号 / 主務所属が確定している。
  - 参照したい申請の識別子が特定されている。
- 入力: 申請の識別子
- 出力: 申請の詳細情報（申請者・休暇種別・期間・申請理由・状態・判断情報・利用者の承認可否フラグ）
- メインフロー
  1. 利用者が申請の識別子を指定して詳細表示を要求する。
  2. システムが対象申請の存在を確認する（BR-PRE-005）。
  3. システムが対象申請の詳細情報を取得する。
  4. システムが対象申請に対する利用者の承認可否を ApprovalPolicy で判定し、詳細情報と合わせて返す（BR-POL-001）。
  5. システムが詳細情報を利用者に提示する。
- 代替フロー
  - なし
- 例外
  a. 対象申請が存在しない場合
  - 業務エラー「指定された休暇申請が存在しません」を提示する。
- 事後処理
  - 参照系であり申請の状態は変更されない。
  - 利用者が承認権者かつ申請が「申請中」の場合、詳細画面から承認 / 却下操作へ遷移可能になる。
