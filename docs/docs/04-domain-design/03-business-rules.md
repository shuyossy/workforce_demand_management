# ビジネスルール台帳

ドメイン / ユースケース起点のルールのみ収録する（横断的な技術ルールは技術設計書 §ロギング / §認証認可 等を参照）。

各ルールには「概要」「詳細参照」「実装場所」「テスト」を明記し、設計段階でも台帳から仕様にたどり着ける形にする。

適用先プロジェクトでは、後述のサンプル例の行を案件のビジネスルールに差し替える運用とする。サンプル例は不要であれば §サンプル例 ごと削除すればよい（→ [サンプル削除ガイド](../getting-started/strip-sample)）。

## 凡例

- **種別**:
  - `不変条件 (INV)`: モデルが常に満たす条件（コンストラクタ / メソッドで検証）
  - `方針 (POL)`: 業務判定の規則。標準実装場所はドメインサービスだが、判断スコープが単一集約内で完結する場合はエンティティ側、ユースケース固有の判定が必要な場合はアプリケーション層に置いてもよい。実装場所はテンプレ表の「実装場所」列で表現する
  - `事前条件 (PRE)`: UseCase 実行時にチェック、違反は `MSTBusinessException`

## 事後条件についての方針

本ボイラープレートは **「ユースケース設計時点で事後条件 (postcondition) は明示的に管理しない」** 方針を採る。状態の整合性は不変条件（INV）と前提条件（PRE）で十分担保でき、列レベルの「何を保存するか」を業務ルールとして並べると認知負荷を上げて実装詳細に寄ってしまうため。

業務上有意な「操作の結果として何が成立するか」は、ユースケース章の **事後処理（業務アウトカム）** で表現する（例: 「業務オブジェクトが受付状態になり一覧に表示される」「操作時点のアクター情報が監査用に保全される」）。実装側のフィールド保存・ログ出力・トランザクション境界等の技術スナップショットは [技術設計書](../technical-design) 側に記載する。

## 分類体系についての方針

本ボイラープレートは **INV / POL / PRE の3分類で十分** という立場を採る。ホーア論理（前提条件・不変条件・事後条件）由来の古典的枠組みであり、ドメインルールの主要カテゴリを網羅する。事後条件は前節のとおり「業務アウトカム」としてユースケース章で吸収する。

適用先プロジェクトでよく追加候補に挙がるカテゴリは、以下の判定ガイドに従って既存3分類に振り分ける。新カテゴリ追加（`BR-XXX-NNN` の新規プレフィックス定義）は、本ガイドで吸収できない案件固有の事情がある場合に限り認める。

### 判定ガイド

| 追加候補                 | どこで扱うか                                                            |
| ------------------------ | ----------------------------------------------------------------------- |
| 状態遷移ルール           | INV（遷移表に従うこと自体）+ PRE（現状態から次状態に遷移可能か）        |
| 監査・スナップショット   | ユースケース章の「事後処理（業務アウトカム）」（前節参照）              |
| クエリ仕様 / 検索条件    | ユースケース章。検索結果が業務判定に影響する場合のみ POL として追加     |
| カレンダー依存ルール     | POL（営業日換算・締切判定はドメインサービスの責務）                     |
| 多段承認 / 代理承認      | POL の拡張（承認可否判定の入力を増やす形）                              |
| アプリケーション固有判定 | POL（実装場所はユースケース層でも可、テンプレ表の「実装場所」列で明示） |

### INV / POL / PRE の境界

- **INV vs POL**: モデル単体で完結する状態ルールは INV、複数の属性・集約・コンテキストから判断する規則は POL
- **POL vs PRE**: 業務上の判断ルール（業務分析者が言語化できるドメイン規則）は POL、ユースケース実行に必要な前提条件（対象存在・入力妥当性等の運用上のチェック）は PRE

## 台帳に含めないルール

横断的技術ルールは技術設計書側に集約しているため、本台帳には含めない：

- BR-POL-002 部長層コード設定化 → [技術設計書](../technical-design) §認証/認可
- 業務イベント INFO ログ → [技術設計書](../technical-design) §ロギング
