# ADR-004 フラットカバレッジ閾値とクラス単位例外運用

## コンテキスト

カバレッジ閾値をパッケージ層別に細かく分けると、適用先プロジェクトで「なぜこの層だけ閾値が低いのか」を読み解くコストが大きくなる。閾値はフラットに統一し、例外は Java クラス単位で台帳管理する方針とする。

## 決定

JaCoCo の `<rule>` を `element=BUNDLE` 1 本でアプリ側パッケージ全体を評価し、本体と `-Pe2e` の 2 系統でフラットな閾値を verify で hard gate する。

| 系統                                       | LINE | BRANCH | 達成手段                     |
| ------------------------------------------ | ---- | ------ | ---------------------------- |
| 本体（`-Pfast` / `-Pci-mr` / `-Pci-main`） | 85%  | 85%    | 単体テスト + 結合テスト      |
| `-Pe2e`                                    | 95%  | 95%    | 単体 + 結合 + E2E カバレッジ |

### クラス単位例外運用

フラット閾値を技術的・構造的に達成できないクラスは、`pom.xml` plugin-level `<excludes>` に列挙し、本 ADR の例外台帳に根拠を残す。

**例外追加の手順:**

1. 構造側で改善できないか先に検討（テスト可能な POJO に分解、ヘルパ抽出、コンテナ依存の局所化など）
2. それでも達成不能であれば `pom.xml` の `jacoco-maven-plugin` `<configuration>/<excludes>` に classes path 形式（例：`jp/mufg/it/rcb/foo/Bar.class`）で追記
3. 本 ADR の例外台帳に「FQCN / 達成不能な理由 / 代替の品質担保策」を 1 行追記
4. `-Pe2e` でも対象外としたい場合は `-Pe2e` プロファイル側の `<excludes combine.self="override">` にも同じクラスを追記
5. `<excludes>` に入れたクラスは HTML レポートからも消える点を了承する（レポートで除外状況を確認したい場合は本 ADR の台帳と `pom.xml` を突き合わせる）

### クラス単位例外台帳

#### 本体（`-Pfast` / `-Pci-mr` / `-Pci-main`）で除外

| FQCN                                              | 達成不能な理由                                                                 | 代替の品質担保策                                               |
| ------------------------------------------------- | ------------------------------------------------------------------------------ | -------------------------------------------------------------- |
| `jp.mufg.it.rcb.shared.web.AuthenticationFilter`  | Servlet Filter は Servlet コンテナでのみ動作するため単体・結合テストで計測不能 | `-Pe2e`（Playwright）で未認証アクセス → リダイレクト経路を網羅 |
| `jp.mufg.it.rcb.shared.web.AccessLogFilter`       | 同上                                                                           | `-Pe2e` の全 HTTP リクエストで MDC put / remove 経路を網羅     |
| `jp.mufg.it.rcb.shared.web.RequestEncodingFilter` | 同上                                                                           | `-Pe2e` の全 HTTP リクエストで UTF-8 設定経路を網羅            |

#### `-Pe2e` でも除外

| FQCN                                        | 達成不能な理由                                                                                                                                                                  | 代替の品質担保策                                                                                                            |
| ------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| `jp.mufg.it.rcb.shared.web.AccessLogFilter` | `accessLogger.isLoggable(Level.INFO)` の false 分岐は、アプリの実行時ログレベルが常に INFO 以上である E2E 実行環境では踏めない（Filter は単体テスト対象外の方針のため代替不可） | ログレベル分岐はパフォーマンス目的のガード（重い String.format 回避）であり、ログ出力自体は全 HTTP リクエストで到達確認済み |

#### パッケージ単位除外（社内ライブラリ準拠 IF）

社内ライブラリ準拠 IF パッケージは内容変更不可・将来差し替え予定のため、いずれの系統でも常に対象外。

| パッケージ                                               | 理由                                                      |
| -------------------------------------------------------- | --------------------------------------------------------- |
| `jp.mufg.it.rcb.config`                                  | 社内ライブラリ準拠 IF。将来本物のライブラリに差し替え予定 |
| `jp.mufg.it.rcb.log.cdi`                                 | 同上                                                      |
| `jp.mufg.it.rcb.log.formatter`                           | 同上                                                      |
| `jp.mufg.it.rcb.exception`（`handler` / `inner` を含む） | 同上                                                      |
| `jp.mufg.it.rcb.userinfo`（`dto` / `context` を含む）    | 同上                                                      |

## 結果

- 閾値の判断材料が「2 系統 × フラット 2 値」に集約され、適用先プロジェクトで迷わない
- 例外は本 ADR 1 ファイルが単一の真実の源になり、追加・削除の根拠が一元管理される
- 計装フェーズで除外したクラスは HTML レポートから消えるため、「テストされていないように見える」という混乱が起きない

## 代替案

- **全クラス一律 60%**: 単純だがドメインロジックの脆弱性を許容するため却下
- **層別閾値（5〜7 層を個別調整）**: 適用先で「層の意味」を理解するコストが大きく却下
- **`element=CLASS` rule で個別下限を強制**: JaCoCo の rule 評価は AND であり、BUNDLE rule の集計から該当クラスを外せないため、現実的には plugin-level `<excludes>` と等価。台帳管理の簡潔さから `<excludes>` 統一を採用
