# アプリ開発の進め方

TDD,DDDを前提とする
1-1. 設計書の修正（`docs`配下）
1-2. 振る舞いを確認するためのテストを作成 2. 1のテストがグリーンになるように実装 3. リファクタリングと、テスト戦略に沿ってテストを拡充

# 開発サイクル

シフトレフトを大前提とする。
本プロジェクトではAIを利用した開発が大前提となるので、git hookに様々な品質チェックを紐づけている。
詳細は `rules/README.md` を参照。

# 制約条件

- 本システムは社内のプライベートなネットワーク環境で利用する前提
  - mavenやnpmの公式リポジトリには接続可能だが、それ以外には接続不可能

# 作業時の注意点

- コードのコメントは日本語で記載すること
- Google Java Style に厳密に沿うこと（git hookでチェックが入る）
- 静的解析を回避するようなアノテーションやコメントをコード内に記載することは禁止（個別抑制が必要な場合は理由を必ずコメントで残すこと）
- 社内ライブラリ準拠 IF パッケージの **内容を変更しないこと**
- 作業完了後はビルドがエラーなく完了することを確認し、commit/pushまで実行すること
- クリーンアーキテクチャの中心部分から外側に向けてコーディングを進めること
- プロジェクト全体を把握して、全ての実装が必要箇所を正しく洗い出してから実装すること
- DRY原則を守ること
- UIは実際に市場に投入できるくらいレベルの高いUIにすること
  - 画面はビジネスライクでプロフェッショナルな印象を与え、全ての画面でスタイル一貫性を保つこと
  - 初心者ユーザでも利用しやすいようにすること。メンタルモデルに反さないように注意すること
- URLについてもベストプラクティスに沿って設計すること
  - ブックマーカブルなアプリケーションになるように意識すること

# コードレビュー観点の一例

- 受け入れ基準を満たしているか
- アーキテクチャ戦略に沿って作成されているか
- 例外は方針に沿って正しく処理され、エラーメッセージは正しくユーザに通知されているか
- 冗長な処理はないか
  - 共通化できていない処理はないか
    - バックエンド、コンポーネントどちらも確認
    - コンポーネントはなるべく共通化して保守性を高められているか（必要に応じてshadcn/uiも導入）
- 将来的なバグにつながるようなコードはないか
- 一覧データ等の初回データ取得はRSCで実施できないか検討できているか
- mastra関連コードは設計方針に沿って実装できているか
- テストコードは問題なく作成できているか
  - 受け入れ基準を検査できているか
  - テスト観点は足りているか
    - ホワイトボックステストをすべき複雑なロジックはないか
    - データのバリエーションテストが必要な箇所はないか
    - 異常系も漏れなく作成できているか
  - 適切にアサーションできているか
    - 検査項目不足が散見されるので、注意すること
